# Player money and equipment acquisition

`PlayerMoneyAccount` is the single player-owned source of currency truth. It stores whole, safe-integer currency units; the initial balance is `500` and the inclusive maximum is `999,999,999`. Both values are exported constants. There are no fractional units or persistence/network claims in this local slice.

## Public account API

- `credit(amount, { reason, source? })` commits a positive integer credit or returns `undefined` at the maximum.
- `debit(amount, { reason, source? })` commits only when affordable. An unaffordable debit returns `undefined` without a balance change or event.
- `canAfford(amount)` performs the same integer validation without mutating.
- `reset(balance?, metadata?)` restores the documented starting balance by default.
- `getSnapshot()` returns a frozen, detached read-only view with balance, maximum, sequence, and the last typed transaction.
- `events.transaction` and `events.balanceChanged` publish the same immutable transaction payload with signed delta and reason/source metadata.
- `dispose()` is idempotent, clears listeners, and makes later reads that validate availability or mutations fail explicitly.

Missions, rewards, pickups, purchases, and any future shop are callers of this API. They must not cache or own a second balance. `MoneyHudSystem` is likewise only an observer. It uses deterministic `en-US` integer formatting, a local-safe condensed/monospace font stack, tabular numerals, a stable-width balance lane, and one signed delta lane. Rapid transactions coalesce algebraically and restart the dismissal window. The CSS disables motion under `prefers-reduced-motion`; listeners, timer, and DOM are all released on disposal.

## Equipment entitlement and purchase policy

`CharacterEquipment` now separates catalog existence from ownership. Its `ownedIds` set is authoritative, `owns()` is the read API, and `equip()` plus quickbar toggles reject unowned definitions. The quickbar retains both fixed slots to avoid layout/input churn, but marks an unowned handgun as locked and hides its ammunition. Generic/NPC construction still defaults to the complete catalog to preserve established NPC fixtures. The composed player deliberately starts with only the knife.

`acquire()` grants without equipping. `acquireAndEquip()` produces a compensation token for transaction-oriented callers, and `rollbackAcquisition()` restores the previous equip state when the surrounding operation cannot commit. These are entitlement operations, not catalog mutations.

`HandgunPurchase` is the minimal purchase policy, not a shop. The local procedural handgun costs `250` units. It checks already-owned and affordability first, calls the equipment acquisition/equip boundary, then debits. A repeated purchase is idempotent; insufficient funds or acquisition failure creates no debit. The compensation token covers the otherwise-unexpected case where the final debit cannot commit. No remote asset is involved: the existing handgun prop is generated by `EquipmentPresentation`, and its native animations remain in the local character GLBs.

## Development fixtures

Development tools expose passive money/pickup state under the existing Player and Interactions groups. Mutations stay in Commands / Actions:

- `player.money-credit [amount]`
- `player.money-spend [amount]`
- `player.money-reset`
- `player.cash-pickup-spawn`
- `player.cash-pickup-remove`
- `player.handgun-purchase`

The cash fixture composes `GameObjectWorld` with `InteractionSystem`. It spawns directly ahead of the player, credits `100` once with metadata, hides immediately after a successful collection, lets the interaction complete, then releases its registration, geometry, and materials. It is development-only and does not introduce inventory, missions, networking, or a shop UI.

The opt-in browser bridge exposes `money.account`, `money.hud`, and `money.cashPickup` snapshots. Browser coverage asserts balance/HUD deltas, unaffordable rejection, pickup completion, purchase deduction/equip, repeat protection, unchanged game state, and clean console/runtime reporting at desktop and narrow sizes.
